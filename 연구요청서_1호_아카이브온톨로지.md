> **배경**: archive-curator 스킬로 문서 분류까지는 성공했으나, "분류된 문서 → 온톨로지 → 결과물 생성"으로 가는 다리가 없음. 팔란티어의 온톨로지 개념을 벤치마크하여, 대량 아카이브에서 지식을 구조화하는 범용 프레임워크를 설계해야 함.

---

## 연구 목적

대량의 비정형 문서 아카이브를 **온톨로지(객체 + 관계 그래프)**로 변환하는 범용 프레임워크를 설계한다. 단순 RAG(문서 잘라서 검색)를 넘어, 지식 간 논리적 관계를 보존하여 의사결정트리, 업무 위저드, 맥락 기반 AI 답변 등 다양한 결과물을 생성할 수 있는 구조를 목표로 한다.

## 연구 범위

### 1. 팔란티어 온톨로지 벤치마크

- Palantir Foundry의 Ontology SDK 접근법: 객체 타입(Object Types), 관계 타입(Link Types), 액션(Actions) 모델링 방식
- 팔란티어가 "도메인 모델을 먼저 정의하고, 데이터를 매핑"하는 프로세스
- 이것이 아카이브(문서 기반)에 적용될 때의 차이점과 적응 방법
- 팔란티어 외 참고할 수 있는 접근법 (Knowledge Graph, Semantic Web, 법률 온톨로지 등)

### 2. 아카이브 유형별 온톨로지 패턴

다음 유형별로 "어떤 객체와 관계가 추출되는가"를 분석:

| 아카이브 유형 | 예시 | 예상 온톨로지 유형 |
|---|---|---|
| 업무규정/방법서 | 여신업무방법서 880p | 프로세스·규제 온톨로지 |
| 연구 논문/실험 결과 | TopoVec 연구 아카이브 | 개념·증거 온톨로지 |
| 기술 문서/코드베이스 | SPARK, DNA 문서 | 시스템·컴포넌트 온톨로지 |
| 법률/컴플라이언스 | 은행 내규, 금융규정 | 권리·의무 온톨로지 |

**핵심 질문**: 유형별로 다른 점은 무엇이고, 공통 프레임워크로 추상화할 수 있는 부분은 어디까지인가?

### 3. 문서 분류 → 온톨로지의 다리

현재 archive-curator가 뽑아내는 것: Category, Tags, Value Grade, AI Summary
온톨로지에 필요한 것: 타입이 있는 객체(Typed Objects), 타입이 있는 관계(Typed Links), 조건 로직(Conditions)

- 이 갭을 메우려면 **추출 스키마**가 어떤 형태여야 하는가?
- 도메인별 추출 스키마 템플릿을 설계할 수 있는가?
- 기존 archive-curator를 확장하는 게 나은가, 별도 스킬(doc-distillery)이 필요한가?

### 4. 저장 구조와 결과물 생성

- 온톨로지를 노션에 저장한다면 어떤 DB 구조가 필요한가? (현재 📂 문서 아카이브는 문서 단위, 온톨로지는 지식 단위)
- V6 메모리(Redis 그래프)와의 연계 가능성
- 하나의 온톨로지에서 생성 가능한 결과물 유형: 의사결정트리, 업무 위저드, 교육 자료, 감사 트레일, AI 맥락 답변 등

## 기대 결과물

1. **아카이브 온톨로지 프레임워크**: 공통 구조 + 도메인별 확장 패턴
2. **추출 스키마 설계안**: 문서에서 객체/관계/조건을 뽑아내는 방법
3. **여신업무 온톨로지 초안**: 업무방법서 구조(편/장/절)를 기반으로 한 객체·관계 모델 예시
4. **아키텍처 제안**: archive-curator → (추출) → 온톨로지 저장 → 결과물 생성 파이프라인
