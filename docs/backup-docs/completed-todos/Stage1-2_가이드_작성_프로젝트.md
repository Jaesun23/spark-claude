## 1. 프로젝트 개요

### 목적
AI 협업 방법론의 **초기 단계 (Stage 1-2)** 가이드를 작성하여, 작은 아이디어를 체계적으로 확장하고 구조화하여 청사진 문서로 만드는 과정을 표준화한다.

### 배경
- ✅ **검증된 하향식 프로세스** (집에서 완성):
  - 청사진 가이드
  - 작업분해 가이드
  - 체크리스트 가이드
  - 구현 단계 (SPARK 시스템)

- ❌ **미완성 상향식 프로세스**:
  - 아이디어 → ??? → 청사진
  - 이 부분의 체계가 없어서 프로젝트 초기에 많은 시행착오 발생

## 2. 문제 정의

### 핵심 문제: "컨텍스트"

7개 프로젝트 분석 결과, 규모가 큰 프로젝트에서 AI 협업 시 다음 문제들이 반복됨:

#### AI의 구조적 한계
1. **맥락 손실**: 프로젝트가 커지면 전체 구조 파악 실패
2. **할루시네이션**: "가능"하다고 했다가 구현 후 "불가능"
3. **똥고집**: 사용자 요구를 무시하고 자기 판단대로 진행
4. **반복 실수**: if-fi, } 누락 등 계속 반복
5. **정보 과부하**: 긴 문서 주면 읽다 말고 임의 행동
6. **새 세션 문제**: 정보 부족 시 엉뚱한 작업 시작

#### 결과
- 리팩토링 반복 (memory-one-spark: 5회, dotfiles: 10회)
- 프로젝트 중단 (6개 프로젝트 실패)
- 완성도 98%인데 가치 0% (memory-one-spark-v5)

### 성공 사례의 공통점

**code-laundry**와 **dotfiles**가 성공한 이유:
- 명확하고 측정 가능한 목표
- 적절한 규모 관리
- AI 컨텍스트 범위 내 통제

## 3. 해결 방향

### 핵심 아이디어: 계층적 템플릿

작은 아이디어를 점진적으로 확장하되, **각 단계를 AI가 처리 가능한 크기로 유지**:

```
아이디어
    ↓ (확장 + 템플릿)
Level 1: 핵심 개념 (1페이지)
    ├─ 핵심 목표
    ├─ 사용자/가치
    ├─ 제약사항
    └─ MVP 정의
    ↓ (구체화 + 템플릿)
Level 2: 주요 영역들 (각 2-3페이지)
    ├─ 영역 A: 기술 선택, 구조
    ├─ 영역 B: 기술 선택, 구조
    └─ 영역 C: ...
    ↓ (상세화 + 템플릿)
Level 3: 상세 결정들 (각 1페이지)
    ├─ ADR-001: 기술 A 선택 근거
    ├─ ADR-002: 아키텍처 B 선택
    └─ ...
    ↓ (통합)
청사진 문서(들)
```

### 핵심 원칙

1. **컨텍스트 관리**
   - 각 레벨은 작음 → AI가 한 번에 처리 가능
   - 레벨 간 독립적 → 맥락 손실 방지

2. **일관성 보장**
   - 템플릿 질문들이 구조 강제
   - 각 레벨마다 동일한 형식

3. **AI 통제**
   - 명확한 지침 → "좋을 대로" 행동 방지
   - 체크박스 → 누락 방지
   - 검증 가능 → 할루시네이션 차단

## 4. 작업 계획

### Phase 1: 실전 재구성 (현재)

**선택한 프로젝트**: memory 시스템

**선택 이유**:
- ✅ **실제 필요성**: 1호, 2호와 대화에 메모리 필요
- ✅ **반복 실패**: V3, V5 모두 아키텍처 위반으로 실패
- ✅ **적절한 규모**: 너무 작지도 크지도 않음
- ✅ **BioNeX 대비 단순**: 챗봇 제외, 메모리만 집중

**방법**:
```
1. 시작점: "Jason이 메모리 시스템 필요"
2. 2호와 대화하며 계층적으로 구체화
3. 각 단계마다 기록:
   - 어떤 질문이 필요했나?
   - 어떤 정보가 필요했나?
   - 어떤 결정을 했나?
   - V3/V5는 여기서 뭘 놓쳤나?
4. 템플릿 항목 도출
```

### Phase 2: 템플릿 정리

재구성 과정에서 발견한 것들을 템플릿으로 정리:
- Level 1 템플릿 (핵심 정의)
- Level 2 템플릿 (영역 분해)
- Level 3 템플릿 (상세 결정, ADR 형식)
- 레벨 간 연결 규칙
- 일관성 검증 체크리스트

### Phase 3: 가이드 문서 작성

템플릿을 기반으로 실용적 가이드 작성:
- 언제 사용하는가?
- 어떻게 사용하는가?
- 주의사항 (AI 한계 고려)
- 실전 예시 (memory 프로젝트)
- 검증 방법

### Phase 4: 검증 및 개선

다른 프로젝트에 적용하며 검증:
- 실패 프로젝트 재분석
- 새 프로젝트 시작 시 테스트
- 피드백 반영 및 개선

## 8. 핵심 인사이트

### Jason의 발견
> "청사진→작업분해→체크리스트→구현"은 검증되었다.
> 문제는 "아이디어→청사진"이다.
> **화두는 "컨텍스트"다.**

### 2호의 이해
> 작은 프로젝트(code-laundry)는 성공한다.
> 큰 프로젝트(memory, dotfiles)는 리팩토링 반복한다.
> **계층적 템플릿으로 AI 컨텍스트를 관리하면 규모 확장이 가능하다.**

### 공통 결론
> **"규모가 커져도 각 단계를 작게 유지하면**
> **AI의 한계를 극복하고 일관성을 유지할 수 있다."**
