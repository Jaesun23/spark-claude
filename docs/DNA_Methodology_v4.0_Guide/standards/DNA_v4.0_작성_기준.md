## 1. 문서 유형별 역할 정의

### 1.1 세 가지 문서 유형

| Type | 코드 | 역할 | 길이 | 독자 |
|------|------|------|------|------|
| **Guide** | G | 간결한 가이드 (지침+템플릿) | 1,000-1,500줄 | 실무 작업자 |
| **Manual** | M | 상세 해설서 (왜? 어떻게?) | 1,500-3,000줄 | 이해 필요자 |
| **Example/Case** | E | 실전 사례집 (전체 흐름) | 500-2,000줄 | 참고 예시 |

### 1.2 역할 상세

#### Guide (G) - 완결된 실행 매뉴얼

**목적**:
- **완결성**: Guide만 봐도 해당 Stage 문서 생산 가능
- **독립 실행**: 명확한 경우는 Guide에 모두 담기
- **선택적 참조**: Manual/Case는 애매하거나 사례 필요시만 (컨텍스트 소비 각오)

**포함 내용** (충분히 담기):
- 필수 입력/출력 문서 목록
- 모든 질문 + **명확한 판단 기준**
- **구체적** 작성 원칙 (충분한 예시)
- 작성 방법 템플릿
- 체크리스트
- Manual/Case 경로 (언제 봐야 하는지 + 컨텍스트 소비 경고)

**제외 내용** (불필요한 것만):
- 이론적 배경 (왜 이 방법론이 나왔는지)
- 역사적 맥락 (CAP 이론의 유래 등)
- 철학적 논의
- 중복 설명

**핵심**: 간결함 ≠ 단순함. 필요한 것은 모두 담되, 불필요한 것만 제거

**비유**: "완결된 레시피" - 재료, 조리법, 팁, 주의사항 모두 포함. 단, 요리의 역사는 제외

#### Manual (M) - 애매한 상황 해설서

**목적**:
- **애매한 경계 판단**: Guide의 명확한 기준으로 부족할 때
- **깊은 이해**: 왜 이렇게 하는지 이론적 배경
- **Trade-off 분석**: 여러 선택지 간 장단점 비교

**사용 시점** (컨텍스트 소비 각오):
- "이해가 도저히 안 된다"
- "A와 B 경계가 애매모호하다"
- "왜 이 방법을 쓰는지 배경 지식 필요"

**포함 내용**:
- 이론적 배경 (CAP 이론, NFR trade-off 등)
- 애매한 경계 케이스 분석
- 각 선택지의 trade-off 상세 비교
- 실패 사례 및 주의사항
- 심화 예시 (복잡한 시나리오)

**제외 내용**:
- Guide와 중복되는 명확한 판단 기준
- 간단한 예시 (Guide에 있어야 함)

**비유**: "요리 교과서" - "이 재료는 왜 쓰나요?", "실패하는 과학적 원인"

#### Example/Case (E) - 실전 사례집

**목적**:
- **전체 맥락 이해**: 실제 프로젝트 Stage 1-9 전체 흐름
- **실전 참고**: "진짜 사례가 필요하다"
- **패턴 학습**: 실제 고민, 실수, 해결 과정

**사용 시점** (컨텍스트 소비 각오):
- "이론은 알겠는데 실전 적용이 막막하다"
- "유사한 프로젝트 사례 보고 싶다"
- "전체 흐름을 한 번에 보고 싶다"

**포함 내용**:
- 실제 프로젝트 전체 진행 과정 (Stage 1-9)
- 각 Stage별 실제 산출물 (파일 내용 포함)
- 결정 과정 및 이유 (왜 A 선택? 고민한 점?)
- 발생한 문제 및 해결 방법
- 회고 (잘한 점, 실수, 배운 점)

**제외 내용**:
- 일반적 원칙 (Guide에 있어야 함)
- 이론 설명 (Manual에 있어야 함)
- 완벽한 정답만 (실수도 보여줘야)

**비유**: "요리 블로그 후기" - 실제 만든 과정, 사진, 실패담, "다음엔 이렇게"

## 2. Guide 작성 기준

### 2.1 필수 구조 (모든 Stage 공통)

```markdown
# Stage{N}. {제목} 가이드

> **목적**: Stage {N} - {한 줄 설명}
> **버전**: v4.0 (2025-XX-XX)

---

## 📚 이 가이드의 구성

DNA 방법론은 2호의 컨텍스트 한계(200K tokens)를 고려하여 문서를 분리합니다:

- **이 문서** (Guide): 질문 템플릿 + 작성 원칙 (간결, 1000-1500줄)
  - **목적**: 빠른 참조, 독립 실행 가능
  - **대상**: 실무 작업 시 필수 참고

- **해설서** (Manual): 왜? 어떻게? 상세 설명
  - **목적**: 깊은 이해, 애매한 상황 해결
  - **위치**: `./{0N}M-{0Y}_*.md`

- **사례집** (Cases): 실전 프로젝트 전체 흐름
  - **목적**: 전체 맥락 이해, 참고 예시
  - **위치**: `./{0N}E-{0Y}_*.md`

**원칙**: Guide만 봐도 작성 가능, Manual/Case는 필요시에만 참고
```

### 2.2 Guide 작성 원칙

#### 원칙 1: 간결성 (1,000-1,500줄)

**목적**: 2호의 컨텍스트(200K tokens) 내에서 전체 Stage 작업 가능

**방법**:
- 핵심만 담기
- 상세 설명은 Manual 참조로 대체
- 중복 제거

**검증**:
```bash
wc -l {0N}G-00_*.md
# 1,000 ~ 1,500 범위 확인
```

#### 원칙 2: 독립 실행 가능

**의미**: Guide만 읽고도 해당 Stage 완료 가능

**필수 포함**:
- 모든 입력 문서 목록
- 모든 출력 문서 목록
- 모든 필수 질문
- 간단한 작성 원칙
- Manual 참조 링크

**검증**:
- Agent가 Guide만으로 작업 완료 가능한가?
- 추가 질문 없이 산출물 생성 가능한가?

#### 원칙 3: 맥락 설명 (Why This Part)

**위치**: 각 Part 시작 부분

**포함 내용**:
1. 이전 Part 요약 (1-2문장)
2. 왜 이 Part가 필요한지 (2-3문장)
3. 이 Part의 목표 (1-2문장)
4. (선택) 다음 Part 예고 (1문장)

**예시**:
```markdown
## Part 1: 시스템의 패밀리 유형 파악

Part 0에서 핵심 기능을 파악했습니다. 이제 이 시스템이 **어떤 종류의 시스템인지** 파악해야 합니다.

DNA 방법론에서는 모든 소프트웨어 시스템을  3가지 핵심 차원으로 분류하여
7가지 패밀리로 구분합니다. 패밀리를 정하면 기술 스택이 자동으로 좁혀집니다.
```

#### 원칙 4: 참조 경로 완전 명시 + 사용 시점 명확화

**목적**:
- Agent/2호가 실제로 참조할 수 있도록 (경로 + 라인)
- **컨텍스트 소비 각오**하고 보도록 (사용 시점 명시)

**형식**:
```markdown
**{사용 시점}**: **{Manual|Case}**(`{경로}:{시작줄}-{끝줄}`) {섹션}
  ({예시 또는 주의사항})
  (주의: {분량} 분량, 컨텍스트 소비 큼)
```

**예시**:
```markdown
**애매한 경계 판단**: **Manual**(`./01M-01_layer1_manual.md:45-120`) 1.2절
  (예: "B2B SaaS는 A인가 B인가?" - 고객 계약 조건에 따라 다름)
  (주의: 75줄 분량)

**실전 적용 사례**: **Case**(`./02E-01_stock_trading_case.md:234-456`) Stage 1
  (주식 거래 플랫폼 전체 결정 과정, 고민과 실수 포함)
  (주의: 220줄 분량, 컨텍스트 소비 큼)
```

**금지** (마치 필수처럼 보이는 표현):
```markdown
❌ **상세 설명**: Manual 1.2절 참고
❌ **자세한 내용**: Manual 참고
❌ **참고**: Case 참고
```

**실행 가능성**:
- Agent가 Read tool의 offset/limit에 즉시 적용 가능
- 사용 시점 명시로 불필요한 컨텍스트 소비 방지

#### 원칙 5: 구조 용어 통일

**Stage - Part - Layer/Question 계층 준수**

| 계층 | 사용 범위 | 명명 규칙 |
|------|----------|----------|
| **Stage** | 전체 (1-9) | Stage 1, Stage 2, ... |
| **Part** | Stage 내부 (0-N) | Part 0, Part 1, Part 2, ... |
| **Layer** | **Part 1 전용** (3-Layer Decision Tree) | Layer 1, Layer 2, Layer 3 |
| **Question** | 각 Part/Layer별 | Q0-1, L1-Q1, P2-Q1 |

**핵심 원칙**:
- **Layer는 Part 1에서만 사용**: 3-Layer Decision Tree는 특수 개념
- **다른 Part는 Layer 사용 금지**: Part 0 = Q0-X, Part 2 = P2-QX
- **일관성 유지**: 모든 Stage에서 동일한 명명 규칙

**검증**:
```bash
# Layer 오용 확인 (Part 1 외에서 Layer 사용하면 안 됨)
grep -n "Layer [0-9]:" 0*G-00_*.md | grep -v "Part 1"
```

## 3. Manual 작성 기준

### 3.1 Manual의 역할

**Guide와의 차이**:
- **Guide**: 완결된 실행 매뉴얼 - "무엇을, 어떻게 하라" (명확한 경우 모두 담김)
- **Manual**: 애매한 상황 해설서 - "왜, 어떤 경계 케이스가?" (컨텍스트 소비 각오)

**Manual이 필요한 경우** (사용 시점):
1. **이해가 도저히 안 된다**: Guide의 설명만으로 개념 이해 불가
2. **애매모호하다**: 여러 선택지 간 경계가 불분명 (A vs B 구분 어려움)
3. **배경 지식 필요**: 이론적 근거 없이는 적용 불가 (CAP 이론, NFR trade-off 등)
4. **실패 사례 확인**: 어떤 함정이 있는지 미리 알고 싶음

**핵심**: Manual은 필수가 아니라 **선택적** - 컨텍스트 소비 각오하고 보는 것

### 3.2 Manual 작성 구조

```markdown
# {0N}M-{0Y}_{주제}_manual.md

## 1. 개요

{이 Manual의 목적과 범위}

---

## 2. 배경 지식

{이론적 배경, 선행 개념}

---

## 3. 상세 설명

### 3.1 {주제 1}

{Guide의 간단한 설명을 확장}

#### 왜 이 방법을 사용하는가?

...

#### 어떻게 적용하는가?

...

#### Trade-off는?

...

---

## 4. 실패 사례 및 주의사항

### 4.1 흔한 실수

...

### 4.2 주의사항

...

---

## 5. 심화 예시

{복잡한 시나리오 예시}

---

## 6. 참고 자료

{관련 문서, 외부 자료}
```

### 3.3 Manual 작성 원칙

#### 원칙 1: Guide 보완, 중복 제거

**DO**:
- Guide에서 "간단히" 설명한 것을 "깊이" 확장
- Guide에서 제공 안 한 배경 이론 추가
- Guide에서 다루기 어려운 복잡한 예시

**DON'T**:
- Guide와 똑같은 질문 템플릿 반복
- Guide에 이미 있는 간단한 예시 재작성

#### 원칙 2: 독립 읽기 가능

**의미**: Manual도 독립적으로 읽을 수 있어야 함

**필수 포함**:
- Manual의 목적 및 범위 (개요)
- 최소한의 맥락 (Guide 참조)
- 자체 완결적 설명

**형식**:
```markdown
## 1. 개요

**목적**: 이 Manual은 {Guide에서 다룬 주제}에 대한 상세 해설입니다.

**대상 독자**: {Guide만으로 부족한 경우}

**관련 Guide**: **{0N}G-00**(`../0{N}G-00_*.md`) 참고
```

#### 원칙 3: 1,500-3,000줄

**이유**: 상세하되, 너무 길어 컨텍스트 초과 방지

**검증**:
```bash
wc -l {0N}M-{0Y}_*.md
# 1,500 ~ 3,000 범위 확인
```

## 4. Example/Case 작성 기준

### 4.1 Case의 역할

**목적**:
- 실제 프로젝트 전체 흐름 보여주기
- Guide/Manual의 이론을 실전에 적용한 모습 보기
- 성공/실패 패턴 학습

**사용 시점** (컨텍스트 소비 각오):
1. **이론은 알겠는데 실전 적용이 막막하다**: Guide/Manual 이해했지만 실제 어떻게 하는지 모름
2. **유사한 프로젝트 사례 보고 싶다**: 내 프로젝트와 비슷한 패밀리/도메인 사례 확인
3. **전체 흐름을 한 번에 보고 싶다**: Stage 1-9가 어떻게 연결되는지 전체 맥락 파악
4. **고민과 실수 과정이 궁금하다**: 완벽한 정답만이 아닌 시행착오 과정 학습

**포함 내용**:
1. 프로젝트 배경
2. Stage 1-9 전체 진행 과정 (또는 특정 Stage 집중)
3. 각 Stage 산출물 (실제 예시)
4. 결정 과정 및 이유 (고민, 실수 포함)
5. 발생한 문제 및 해결

**핵심**: Case는 필수가 아니라 **선택적** - 분량이 많으므로 컨텍스트 소비 각오하고 보는 것

### 4.2 Case 작성 구조

```markdown
# {0N}E-{0Y}_{프로젝트명}_case.md

## 프로젝트 개요

**프로젝트명**: {예: 주식 거래 플랫폼}
**패밀리**: {A-C-A}
**기간**: {2주 계획}

---

## Stage 1: 핵심정의

### 입력
{프로젝트 아이디어}

### Part 0: 핵심 기능 파악

**Q0-1. 이 시스템은 무엇을 하기 위한 시스템인가요?**
```

### 4.3 Case 작성 원칙

#### 원칙 1: 실전 중심

**DO**:
- 실제 발생한 문제 기록
- 실제 고민한 trade-off 설명
- 실제 선택한 이유 명시

**DON'T**:
- 이론적 설명 (Manual로)
- 완벽한 정답만 제시
- 과정 생략하고 결과만 제시

#### 원칙 2: 전체 흐름 보여주기

**의미**: Stage 1-9 전체를 다루기

**이유**:
- Guide/Manual은 Stage별로 분리
- Case는 전체 연결 고리 보여주기

**예외**:
- 특정 Stage만 집중한 Case (예: `03E-02_kent_beck_bplustree_case.md`)
- 제목에 명시 (예: "Stage 3 전용 사례")

#### 원칙 3: 500-2,000줄

**이유**:
- 너무 짧으면: 전체 흐름 파악 어려움
- 너무 길면: 컨텍스트 초과

**검증**:
```bash
wc -l {0N}E-{0Y}_*.md
# 500 ~ 2,000 범위 확인
```

## 5. 공통 작성 원칙

### 5.1 파일명 규칙

**형식**: `{Stage}{Type}-{Seq}_{descriptive_name}.md`

```
01G-00_core_definition_guide.md
│││ ││ └─ 설명적 이름 (snake_case)
│││ │└─ 순서 (00-99)
│││ └─ 구분자 (하이픈)
││└─ 문서 유형 (G/M/E)
│└─ Stage (01-09)
└─ 2자리 숫자
```

**검증**:
```bash
# 파일명 패턴 확인
ls *.md | grep -E '^[0-9]{2}[GME]-[0-9]{2}_[a-z_]+\.md$'
```

### 5.2 버전 관리

**필수 포함**:
```markdown
> **버전**: v4.0 (2025-XX-XX)
> - v4.0 (2025-XX-XX): {변경 내용}
> - v3.0 (YYYY-MM-DD): {변경 내용}
```

**원칙**:
- 구조 변경 = Major 버전 (v3.0 → v4.0)
- 내용 보완 = Minor 버전 (v4.0 → v4.1)
- 오타 수정 = Patch (기록 안 함)

### 5.3 참조 링크 작성

**형식**:
```markdown
**{참조 유형}**(`{상대경로}:{시작줄}-{끝줄}`) {섹션/절 번호} 참고
```

**예시**:
```markdown
**Manual**(`./01M-01_layer1_manual.md:45-78`) 1.2절 참고
**Guide**(`../01G-00_core_definition_guide.md:234-289`) Part 1 참고
**Case**(`./02E-01_stock_trading_case.md:500-650`) Stage 2 예시 참고
```

**실행 가능성 검증**:
```python
# Agent가 이렇게 사용할 수 있어야 함
file_path = "./01M-01_layer1_manual.md"
offset = 45
limit = 78 - 45 + 1
Read(file_path, offset=offset, limit=limit)
```

### 5.4 코드 블록 작성

**형식**:
````markdown
```python
# 코드 예시
def example():
    pass
```
````

**원칙**:
- 언어 지정 필수 (python, bash, markdown, json 등)
- 실행 가능한 코드 제공
- 주석으로 설명 추가

### 5.5 체크리스트 작성

**형식**:
```markdown
## ✅ 최종 체크리스트

- [ ] {필수 작업 1}
- [ ] {필수 작업 2}
- [ ] {선택 작업 1} (선택)
```

**원칙**:
- 필수/선택 명확히 구분
- 실행 가능한 구체적 항목
- 순서대로 나열

## 6. 검증 체크리스트

### 6.1 Guide 검증

**구조 검증**:
- [ ] 제목 형식: `Stage{N}. {제목} 가이드`
- [ ] 버전 정보 있음
- [ ] "📚 이 가이드의 구성" 섹션 있음
- [ ] "📥 입력 문서" 섹션 있음
- [ ] "📤 출력 문서" 섹션 있음
- [ ] "🔄 다음 Stage로 전달되는 것" 섹션 있음
- [ ] "📍 전체 프로세스에서의 위치" 섹션 있음
- [ ] "✅ 최종 체크리스트" 섹션 있음

**내용 검증**:
- [ ] Part 0, 1, 2... 명확히 구분
- [ ] Layer는 Part 1에서만 사용
- [ ] **완결성**: Guide만으로 해당 Stage 문서 생산 가능 (명확한 경우 모두 담김)
- [ ] **Manual/Case 참조**: "사용 시점" + 경로 + 라인 + 컨텍스트 소비 경고 포함
- [ ] **간결함 ≠ 단순함**: 불필요한 것 제거, 필요한 것 충분히 담김
- [ ] 1,000-1,500줄 목표 (완결성이 우선, 분량은 참고)

**자동 검증**:
```bash

grep -n "Manual\|Case" {0N}G-00_*.md | grep -v ":\|주의:"
```

## 📌 요약

### 핵심 원칙 5가지

1. **역할 분리**:
   - **Guide**: 완결된 실행 매뉴얼 (명확한 것 모두 담기)
   - **Manual**: 애매한 상황 해설서 (컨텍스트 소비 각오)
   - **Case**: 실전 사례집 (전체 흐름, 컨텍스트 소비 각오)

2. **완결성 우선, 간결함 추구**:
   - Guide: 1,000-1,500줄 목표 (완결성이 우선)
   - Manual: 1,500-3,000줄 목표
   - Case: 500-2,000줄 목표
   - **간결함 ≠ 단순함**: 불필요한 것 제거, 필요한 것 충분히

3. **독립 실행**: Guide만으로 해당 Stage 문서 생산 가능

4. **참조 명시**:
   - 사용 시점 + 경로:라인 + 컨텍스트 소비 경고
   - Manual/Case는 선택적 (필수 아님)

5. **구조 통일**: Stage - Part - Layer/Question 계층 준수

### 작성 순서 권장

1. **Guide 먼저**: 핵심 지침 + 템플릿
2. **Manual 다음**: Guide에서 부족한 부분 보완
3. **Case 마지막**: 실전 적용 사례 기록
