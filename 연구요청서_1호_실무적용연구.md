## 조사 범위

### 1. AI 기반 문서 자동 분류 패턴

LLM이 문서를 분류할 때 효과적으로 작동하는 패턴을 조사해줘:

- **프롬프트 기반 분류**: LLM에게 문서를 읽히고 분류하게 할 때의 베스트 프랙티스
  - 분류 기준을 어떻게 제시하면 일관성이 높은가?
  - 한 번에 분류 vs. 다단계 분류(coarse → fine)의 장단점?
  - 문서가 너무 길어서 전체를 읽을 수 없을 때의 전략 (요약 → 분류?)
- **배치 처리 전략**: 수백 개 파일을 효율적으로 처리하는 방법
  - 파일 유형별 전처리 (PDF → text, DOCX → text 등)
  - 토큰 효율성: 전체 문서를 읽을 필요 없이 분류하는 방법은?
  - 분류 결과의 일관성을 보장하는 방법 (동일 문서 = 동일 분류)

### 2. 디지털 아카이브 도구들의 접근법

실제 운영되는 디지털 아카이브/문서관리 시스템이 어떤 메타데이터와 분류 구조를 쓰는지 조사해줘:

- **Notion / Obsidian / Confluence**: 지식 베이스 도구들의 문서 조직 패턴
- **기업 ECM (Enterprise Content Management)**: SharePoint, Box, Google Drive의 분류 체계
- **디지털 아카이브 시스템**: DSpace, Omeka 등 학술/도서관 시스템의 메타데이터 구조
- **개인 지식관리 (PKM)**: Zettelkasten, PARA, Johnny Decimal 같은 개인 파일 정리 방법론

**각 도구/방법론에 대해 알고 싶은 것:**
- 메타데이터 필드는 어떤 걸 쓰는가?
- 분류 체계의 깊이(계층 수준)는?
- AI 자동 분류와 잘 결합되는 구조는?

### 3. 우리 프로젝트에 특화된 고려사항

세 프로젝트의 구체적 상황을 감안한 실전 전략:

#### SPARK / TRAITS (연구·방법론 아카이브)
- 버전 히스토리가 있는 문서의 분류: v1.0 → v4.3 같은 진화 과정을 어떻게 표현?
- 의사결정 맥락(ADR 같은)을 메타데이터에 어떻게 포함?
- 아카이브된 자료와 현재 활성 자료의 구분

#### CURATOR (실무 지식 베이스)
- 규정/매뉴얼 문서: 버전과 유효기간 관리
- 양식(Form) 문서: PDF/DOCX/HWP 동일 양식의 중복 처리
- 업종별 심사 자료: 시간축(2025 H2, 2026 H1)과 업종축의 2차원 분류
- 실제 케이스 스터디: 기업명, 평가일자, 평가유형 등의 메타데이터

### 4. 노션 📁 파일 카탈로그 DB와의 연동

현재 Jason의 노션 파일 카탈로그 DB 구조:

- 이 구조를 확장하여 문서 분류 결과를 저장하려면 어떤 필드가 필요한가?
- 기존 catalog-audit 스킬과의 역할 분담은?
